# Building Polydeukes in Public #1 — A covenant a coding agent can't quietly disable

**English** · [한국어](./2026-07-v0.1-covenant-core.ko.md)

> _v0.1 MVP · the enforcement core lands, guards its own source on the tool and Bash axes, and measures every time it fires._

Yes — this is another entry in the AI-coding-harness space, and 2026 has not been kind to that genre. Engineers report decision fatigue from agent tooling faster than the tooling ships, and most of what ships is prose: rules files, spec templates, instructions the agent may or may not honor. I build with these agents daily and feel the same fatigue, so this series won't ask you to adopt anything. It makes one engineering claim and shows the evidence: **an enforcement layer for coding agents only counts if (a) the agent it constrains cannot quietly switch it off — because the layer protects its own substrate — and (b) it measures its own cost, not just the model's output.** v0.1 of Polydeukes is the smallest system that does both, dogfooded on its own repository.

## The landscape we're building into (and the two gaps in it)

First, what we did *not* invent. Over the past year, vendors, model labs, and academia converged — largely without citing each other — on the same discipline invariants for long-running coding agents: write the verification contract *before* the code; never let the generator grade its own work; externalize state instead of trusting context; give each unit of work fresh context; serialize writes and parallelize only reads. You can find the same list, in different vocabulary, in Anthropic's engineering post on effective harnesses for long-running agents, in Factory's Missions architecture notes, and in the ABC paper's contract-synthesis analysis (arXiv 2602.22302). Polydeukes implements all five in its operating loop, and claims credit for none of them. The interesting part is what the converged techniques still leave open. Two gaps, stated as engineering problems:

**Gap A — enforcement the agent can bypass.** By category, not by name: *rules-as-prompts* (project rules files injected into context) are probabilistic — the agent honors them 70-something percent of the time, which is a compliance rate, not a guarantee. *Spec-driven flows* — the loudest current paradigm — draw the loudest current critique: that they generate "the illusion of work, a bunch of text." The output is prose the agent may or may not follow; a spec is not an enforcement mechanism. *Sandbox guardrails* do enforce — but 2026 bypass write-ups keep documenting agents that manipulate their own settings, and patches that get walked around. The open problem underneath all three: the constrained agent can reach the constraint. (A mechanism-level comparison against specific products needs a per-product teardown; that's deliberately deferred to a documentation page after v1.0.)

**Gap B — nobody measures the enforcement layer itself.** Agent observability and guardrail-metrics products grade the *model's output*: quality, compliance, hallucination. None of the tools we surveyed treat the discipline layer's own behavior — what it blocked, what it passed, when it was bypassed — as a first-class metric. Which means if your enforcement costs more than it protects, you have no way to know.

These two gaps are the seats Polydeukes sits in. The rest of this post is the v0.1 evidence: one hypothesis per gap, each tested on the project itself.

## What v0.1 actually ships

Three pnpm workspaces on one thin protocol:

- **core** — the covenant protocol (stdin JSON in, exit code out; exit 2 blocks), the config loader with protected-path normalization, and ROI telemetry: every judged call appends exactly one record — `blocked`, `passed`, or `bypassed` — to a local log.
- **covenant** — the judges. On the tool axis, a self-mod meta-covenant that blocks Edit/Write calls touching the enforcement layer's own source. On the Bash axis, a shell-mod meta-covenant with heredoc-aware multi-line analysis, write-detection for redirects, `tee`, and `sed -i`, and a read-only allowlist for first tokens like `cat` and `grep`.
- **adapter-claude-code** — up-translates Claude Code's PreToolUse payloads into the covenant input IR, and computes the virtual post-state of Edit/Write/MultiEdit calls without touching disk, so judgment happens before the mutation exists.

Scope boundaries, stated plainly: this is enforcement plus measurement. It is not an autonomy framework, not a linter, and not a spec format.

## Hypothesis 1 — "enforcement must survive the agent it constrains"

**The bet.** A guard the constrained agent can disable is not a guard; it's a suggestion with extra steps. So the covenant must protect its own substrate — recursively: the judges' source, their built output, and the hook wiring that connects them are all protected paths.

**The challenge.** Dogfooding a guard that isn't yet guarded is a contradiction, and a partial guard is barely better. Protect the tool axis but leave Bash open, and the agent disables everything with one line — `sed -i 's/exit 2/exit 0/' covenant/src/self-mod.ts` — logged as nothing at all. We had audited exactly this failure in a predecessor harness: its self-protection was *tool-shaped*, checking Edit and Write while the most common bypass vector ran through the shell. So both axes had to close *before* self-dogfooding turned on, which they did on 2026-07-14. Since then, every Edit/Write/Bash call in this repository's development has passed through the project's own covenants.

**The structural payoff.** The escape hatch turned out to be human-only *by construction*, not by policy. The sanctioned bypass valve is an environment variable; the only path that arms it is Claude Code's settings file — which is itself a protected path. An inline `VAR=1 cmd` doesn't help the agent, because that variable exists only inside a command the hook has already judged. So the agent cannot self-authorize a bypass. Arming is an out-of-session human edit, and every call that rides through an armed hatch is recorded as `bypassed` — never silent. The first dogfooding ticket verified this end to end: edits to `core/src` stayed blocked until a human armed the valve, and the telemetry shows the moment it opened.

**Scope honesty, and a commitment.** Today this human-only property is realized through one runtime. It depends on a Claude Code fact: the settings surface that arms the valve is itself protectable. That is an adapter-specific binding, not yet an agent-neutral guarantee. What generalizes is the *principle* — only a human, outside the session, can arm the bypass. So every future adapter (Codex, pi, whatever comes) carries a designed obligation: find that runtime's equivalent of "the arming surface is itself protected" and build its own human-only arming path on it. An adapter that can't establish this property is a defective adapter, and the acceptance criteria for adapter tickets will say so. This is stated here in public, in advance, so it reads as an obligation rather than a gap we hid.

## Hypothesis 2 — "deterministic gates beat probabilistic checks — but only if you measure them"

**The bet.** An exit-code judgment with a fail-closed default — *cannot judge = block* — is a stronger guarantee than an LLM reviewer or a prompt-level rule. Probabilistic checks have compliance rates; the ABC paper measured five to seven soft contract violations per session even in disciplined pipelines. A deterministic gate either fires or it doesn't, and either way it leaves a record.

**Confirmed — by the machine catching what we missed.** When the last v0.1 ticket merged, the backlog was empty and the milestone looked done. The closing audit ran real commands against the Bash axis instead of trusting the green checkmarks — and caught **five confirmed bypasses** the humans had missed: parent-directory operations (`rm -rf` on an ancestor of a protected tree matched nothing), quote-split paths (`core/sr"c"/…` fell out of routing before the judge ever saw it), backslash-newline splits, a hook catch-all that blocked correctly but wrote zero telemetry rows (silently undercounting the very metric this post cites), and a public export whose judgment boundary had drifted from the judge's. Determinism found what eyes didn't. Then it took **three review passes** to converge the fix — each pass caught a real regression the previous fix introduced. That story is the log below; the short version is that the discipline caught its maker's mistakes three times in a row.

**Challenged — by the cost showing up in our own telemetry.** Deterministic gates aren't free, and if you don't measure them you're flying blind on your own tool. Here is the v0.1 milestone ledger, from the same telemetry the covenant writes:

| verdict | count |
|---|---|
| blocked | 13 |
| passed | 455 |
| bypassed | **552** |

Bypassed went from 56 on the first dogfooding ticket to 552 by milestone close — a 10× jump. The mechanism is structural: the hatch is session-global, so when a human arms it for a ticket that legitimately edits covenant source, *everything else in that window* — a review agent's `git diff`, a `grep`, a `node -e` probe — rides through as `bypassed`. The guard is off precisely when the most sensitive code is being touched, and the telemetry can't distinguish the intended edits from the traffic that coasted through alongside them. We didn't hide that number; we let it reorder the roadmap — the TTL waiver moved to the top of v0.2. That is Gap B answered on ourselves: the enforcement layer's own cost, measured, published, and acted on.

## Where the first goal met resistance — a solved-problem log

Four problems, told as *what we believed → what actually happened → what we changed*. Each one is portable to any harness you might build.

**1. Substring path matching leaks in both directions.** We believed the path-mention + fail-closed principle was enough, with `value.includes(path)` as the mention test. But a substring check knows neither path hierarchy nor shell semantics. Downward: a parent-directory target slips through, because `core` does not *contain* `core/src` — so `rm -rf` on the parent of a protected tree matched nothing. Sideways: the shell removes quotes before executing, but the matcher saw the raw string, so a quote-split path had no contiguous match. We replaced the primitive with segment-aware matching over tokenized (quote-removed) words, and made tokenization failure block rather than fall back to substring. The transferable lesson: a sound principle standing on a leaky primitive is a leaky principle.

**2. The upgrade broke absolute paths — and our own measurement hid it.** We believed the segment fix was verified, because we probed it through the assembled hook. But we probed with *relative* paths, and Claude Code sends `file_path` as an *absolute* path. The new matcher required the protected segments to sit at index 0, so every real edit payload missed — the old substring check had been *accidentally* correct on absolute paths, and the upgrade meant to strengthen protection had disabled it on the primary axis. A review pass caught it before merge. The fix: descendant matching became offset-free. The lesson we now apply everywhere: **measure with representative payload shapes.** Probing with an input the runtime never sends is indistinguishable from not probing at all.

**3. The re-fix over-blocked, and the rule converged on an asymmetry.** Chasing absolute-path *ancestors* too, we loosened the ancestor test to "any suffix of the candidate matches a prefix of the protected path" — and the next review confirmed it now swallowed unrelated directories whose tails happened to collide with a protected prefix. Third pass, final rule: **descendant = contiguous subsequence at any offset; ancestor = root-anchored prefix only.** The cost — absolute-path ancestor operations fall outside deterministic coverage — is accepted explicitly, because the alternative was permanently blocking legitimate work, and over-blocking guards don't get fixed, they get deleted. Complete coverage was never the goal; the two match directions simply need different offset rules, and forcing them to share one guarantees one of them is wrong.

**4. The hatch is hot-arm, cold-disarm.** Arming the bypass via the settings file takes effect immediately; *removing* it does not — the value is already injected into the live session's environment, and deleting its source can't unset a running process's env, so disarming requires a session restart. This is structural friction, not a bug, but it lengthens every arming window — which is exactly why `bypassed` inflates, and exactly why the v0.2 waiver design has expiry built in instead of trusting a human to remember to disarm.

*One aside, in the spirit of this series: this very post is written under the covenant it describes. The mention judgment scans whole payloads, so a document whose body spells out full protected paths gets blocked no matter where the file lives — which is why every path in this post is abbreviated (`core/src` and friends). We report the workaround rather than quietly routing around it; a bypass culture starts small.*

## What v0.1 proved, and what it only suggested

**Proved.** Both-axis self-protection is buildable and deterministic — with the hatch off, edits and shell writes against protected paths exit 2, verified by live probes at the milestone gate. The enforcement layer can measure its own cost — 1,020 judged calls, each leaving exactly one record, with the one silent-undercount bug found and fixed by the audit. And the audit-and-converge discipline catches the maker's own errors — five bypasses and two fix-induced regressions were all caught by the project's own review gates, not by luck.

**Only suggested — with the falsification conditions we're holding ourselves to.** Whether deterministic gates stay net-positive at scale: the `bypassed` ratio is the metric, and if it doesn't collapse after the TTL waiver replaces the session-global hatch, the exit-code-superiority claim gets re-examined, in public. Whether the covenant input IR truly generalizes: one adapter proves nothing — the test is a third agent's payloads passing the same judges through up-translation alone, and that's a v1.0 gate. Whether the ledger loop actually changes behavior: v0.2 exists to measure that, so post #2 will claim it or retract it.

## Next — v0.2, and the one thing the data already changed

v0.2 is the ledger: completion authority moves from the session saying "I'm done" to the harness saying "the verification actions passed" — commands executed by a runner, exit codes as the verdict, runner-owned result fields the agent cannot write. That's invariant number two from the landscape section (generator ≠ grader), hardened from a convention into a mechanism.

But the first ticket of v0.2 was chosen by the data, not the roadmap. The plan had the TTL waiver as a nice-to-have; `bypassed = 552` made it debt. A time-boxed waiver with built-in expiry replaces the session-global hatch, so arming the valve stops meaning "protection is off until someone remembers to restart." Post #2 gets to claim exactly one thing on this front: whether that number collapsed.
